Use the `/payment-links` endpoint to generate a hosted payment page. Instead of building a custom checkout UI, you redirect the payer to a HelloPay-hosted URL where they choose from the payment methods enabled for your account.

You can keep the link fully open, or optionally send a default `rail` and/or `inlineCustomer` when your flow already knows the payer or preferred payment method.

## How it works

1. Send a `POST` request to `/payment-links` with the amount, your internal reference, a `callbackUrl`, and any optional defaults for the checkout flow.
2. HelloPay returns a `paymentLinkUrl` â€” the full URL to redirect the payer to.
3. Redirect the payer to the `paymentLinkUrl`.
4. The payer selects their preferred payment method and completes the flow on the HelloPay-hosted page.
5. HelloPay fires a webhook event when the payment link is completed or expires.
6. Once the flow ends (whether successful or not), the payer is redirected to the `callbackUrl` you provided.

## Redirecting the payer

The `paymentLinkUrl` in the response is the full URL of the hosted payment page. Redirect the payer to this URL as soon as you receive it.

```json
{
  "paymentLinkId": "4a8f2e1b-7c3d-4e5a-9b6c-8d1f2e3a4b5c",
  "paymentLinkUrl": "https://pay.hellopay.com.co/link/V1JKU0tQNU02Wklp5lf2gAY8DYXizUKPzcG39x4i_A",
  "createdAt": "2026-04-20T10:00:00.000Z",
  "expiresAt": "2026-04-20T11:00:00.000Z"
}
```

The page will display the payment methods enabled for your merchant account. If you send `rail`, HelloPay uses it as the default payment method in the checkout, while the enabled payment methods remain available. If you send `inlineCustomer`, HelloPay reuses that customer data when creating the payin from the link.

## Choosing the right flow

`rail` and `inlineCustomer` are optional. Include them only when they match the experience you want to build:

| Flow | Fields to send | When to use it |
| --- | --- | --- |
| Open checkout | Omit both fields | The payer should choose the payment method and enter their own details. |
| Suggested payment method | Send `rail` only | You want the checkout to start on `PSE` or `BRE_B`, but still use the payer-entered customer data. |
| Pre-filled customer | Send `inlineCustomer` only | You already collected customer details and want to avoid asking for them again. |
| Preset checkout | Send both fields | You already know the payer and preferred payment method for the order. |

<Note>
  `rail` is a checkout default, not a payment-method restriction. The available payment methods are still determined by your merchant configuration.
</Note>

<Warning>
  Payment links expire after 1 hour. The `expiresAt` field tells you exactly when. Do not attempt to reuse a link after it has expired â€” create a new one instead.
</Warning>

## Handling webhook events

HelloPay fires payment-link webhook events when a link is completed or expires. You must have a webhook configured to receive these â€” see [Webhooks](/getting-started/webhooks) for setup instructions.

Listen for the following events:

| Event | `data.status` | Meaning |
| --- | --- | --- |
| `paymentlink.completed` | `COMPLETED` | The payer completed the payment successfully. |
| `paymentlink.expired` | `EXPIRED` | The payment link expired before the payer completed it. |

### Sample `paymentlink.completed` payload

```json
{
  "event": "paymentlink.completed",
  "resource": "paymentlink",
  "data": {
    "paymentLinkId": "4a8f2e1b-7c3d-4e5a-9b6c-8d1f2e3a4b5c",
    "reference": "XYZ",
    "amount": 1000,
    "currency": "COP",
    "status": "COMPLETED",
    "createdAt": "2026-04-19T14:32:15.300Z",
    "updatedAt": "2026-04-19T14:32:20.395Z",
    "expiresAt": null,
    "completedAt": "2026-04-19T14:32:20.393Z"
  }
}
```

### Sample `paymentlink.expired` payload

```json
{
  "event": "paymentlink.expired",
  "resource": "paymentlink",
  "data": {
    "paymentLinkId": "4a8f2e1b-7c3d-4e5a-9b6c-8d1f2e3a4b5c",
    "reference": "XYZ",
    "amount": 1000,
    "currency": "COP",
    "status": "EXPIRED",
    "createdAt": "2026-04-19T14:32:15.300Z",
    "updatedAt": "2026-04-19T15:32:15.300Z",
    "expiresAt": "2026-04-19T15:32:15.300Z",
    "completedAt": null
  }
}
```

## Knowing when a payment was made

There are two equivalent ways to find out that a payer has successfully paid against a payment link. Pick whichever fits your integration â€” you don't need to listen to both:

- **`paymentlink.completed` webhook event** â€” fired once the payment link is settled with `data.status` set to `COMPLETED`. Use this if you only care about the payment link lifecycle.
- **`payin.confirmed` webhook event** â€” fired on the underlying payin when it is confirmed. Use this if you already have payin webhook handling in place, or if you want richer details about the specific payin (payment method, rail-level identifiers, etc.). See [Webhooks](/getting-started/webhooks) for the payin event catalog.

Both events carry the `reference` you passed when creating the payment link, as well as the `paymentLinkId`, so you can correlate them back to your order.

## Correlating a webhook event to a payment link

Each webhook event payload includes a `data` object with the payment link details. Use either of these two fields to identify which payment link the event belongs to:

- **`data.reference`** â€” matches the `reference` you passed when creating the payment link.
- **`data.paymentLinkId`** â€” matches the `paymentLinkId` returned in the creation response.

```javascript
// Example webhook handler
app.post('/webhook', (req, res) => {
  const { event, data } = req.body;

  if (event === 'paymentlink.completed' || event === 'paymentlink.expired') {
    const { reference, paymentLinkId, status } = data;
    // status is 'COMPLETED' for paymentlink.completed, 'EXPIRED' for paymentlink.expired
    // Use reference or paymentLinkId to look up the order in your system
    console.log(`Payment link ${paymentLinkId} (ref: ${reference}) is now ${status}`);
  }

  res.sendStatus(200);
});
```

## Callback redirect

After the payment flow ends â€” regardless of whether the payment succeeded or failed â€” HelloPay redirects the payer back to the `callbackUrl` you provided when creating the link. Use this URL to send the payer to a confirmation, error, or status page in your application.

<Note>
  The redirect to `callbackUrl` happens on the client side at the end of the payment flow. It is not a substitute for webhook events â€” always rely on webhooks for authoritative payment status updates.
</Note>

## Request fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `amountType` | `string` | Yes | The only value currently supported is `FIXED`, which means the `amountInCents` you provide is the only amount the payer can pay. |
| `amountInCents` | `number` | Yes | Amount to collect in cents (COP). |
| `reference` | `string` | Yes | Your internal reference for this payment. Returned in payment-link webhook events. |
| `callbackUrl` | `string` | Yes | URL to redirect the payer to after the payment flow ends. |
| `rail` | `string` | No | Default checkout rail. Supported values are `PSE` and `BRE_B`. This does not restrict the payment methods available to the payer. |
| `inlineCustomer` | `object` | No | Customer information to reuse when creating the payin from this link. If omitted, the payer enters customer details in the checkout. |

<Note>
  `amountType` currently only supports `FIXED`. Additional amount-handling modes may be introduced in the future.
</Note>

## Sample request

```bash
curl --request POST \
  --url https://api.hellopay.com.co/payment-links \
  --header 'Authorization: Bearer <token>' \
  --header 'Content-Type: application/json' \
  --data '{
    "amountType": "FIXED",
    "reference": "INV-2026-001",
    "amountInCents": 100000,
    "callbackUrl": "https://hellopay.com.co/callback",
    "rail": "PSE",
    "inlineCustomer": {
      "name": "John Doe",
      "idType": "CO_CC",
      "idNumber": "1000000001",
      "email": "john.doe@example.com",
      "phone": "+573001234567"
    }
  }'
```

In this example, `rail` and `inlineCustomer` are included to preconfigure the checkout. You can omit either or both depending on your flow.
